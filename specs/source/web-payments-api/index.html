This document outlines how to initiate and complete a payment using a polyfill-able browser API.

There are a number of ways that one may participate in the development of this specification:

  • Ad-hoc technical discussion primarily occurs on the public community mailing list: public-webpayments@w3.org
  • Public Web Payments Community Group teleconferences are held on Wednesdays at 1600UTC every other week.
  • Specification bugs and issues should be reported in the issue tracker if you do not want to send an e-mail to the public-webpayments mailing list.
  • Source code for the specification can be found on Github.

Introduction

This API enables a web application to initiate payment for a product or service by calling a navigator.payment.request() method. The implementation of this feature is expected to be provided by a JavaScript polyfill at first, and if deployment is successful, native browser implementations will eventually surface to enhance usability and transaction security.

How to Read this Document

This document is a detailed specification for an application programming interface (API) for initiating payments from within a browser environment. The document is primarily intended for the following audiences:

  • Software developers who want to understand the design decisions and algorithms behind the API.
  • Software developers who want to implement the API.

Payment Flow Overview

A web app running on the merchant's website will interact with a payment service provider via navigator.payment.register(), navigator.payment.request(), and navigator.payment.acknowledge(). Results of the payment request will be POSTed to the merchant's server.

  1. The Web app initiates a payment by calling navigator.payments.request(request).
  2. This starts the flow by sending a purchase request to the payment service provider.
  3. An interface is provided by the payment service provider.
  4. The payer is authenticated by the payment service provider.
  5. The payer accepts or cancels the purchase request.
  6. A callback is called on the merchant's page that notfies the merchants software that the purchase was successful, or canceled.
  7. The merchant software receives a proof of acknowledgement by the payment service provider and may then verify whether or not the payment was processed successfully.

Payment Instrument Registration

  1. The payer navigates to a payment service provider website to sign up.
  2. During sign-up, the navigator.payment.register() call is made to register a payment instrument: 
    var card = {
  '@context': 'https://w3id.org/web-payments/v1',
  scheme: 'https://w3id.org/payment-schemes#Visa',
  label: 'ExampleBank Visa Card'
  paymentRequestService: 'https://pay.example.com/services/cards',
}

// register the payment instrument - at this point, the interface asks the 
// person for approval to store the new payment instrument
navigator.payment.register(card, function(success) {
  if(success) {
    // card has been registered
  } else {
    // card has not been registered
  }
});
    Multiple types of payment instruments may be supported using this approach: 
    var bitcoin = {
  '@context': 'https://w3id.org/web-payments/v1',
  scheme: 'https://w3id.org/payment-schemes#Bitcoin',
  label: 'BitExample Bitcoin Account'
  paymentRequestService: 'https://pay.example.com/services/bitcoin',
}
navigator.payment.register(bitcoin, function(success) {
  if(success) {
    // instrument has been registered
  } else {
    // instrument has not been registered
  }
});

Processing a Payment Request

  1. The payer navigates to a merchant's website.
  2. The payer finds an item to purchase and initiates the payment flow (by clicking a button, swiping, speaking a voice command, etc.).
  3. The merchant application generates a signed purchase request, which is an [[!JSON-LD]] object that contains enough information to process the payment: 
    var req = {
  '@context': 'https://w3id.org/web-payments/v1',
  type: 'PaymentRequest'
  description: 'Payment to ExampleMerch for widgets'
  acceptedScheme: {
    scheme: 'https://w3id.org/payment-schemes#Visa'
    monetaryRequest: {
      amount: '4.35',
      currency: 'USD'
    },
    destination: '20389472398'
  },
  paymentCompleteService: 'https://merchant.example.com/services/paymentComplete?transaction=923847298'
  signature: {
    type: 'GraphSignature2015',
    creator: 'https://merchant.example.com/keys/23',
    created: '2015-09-23T20:21:34Z',
    signatureValue: 'OGQzNGVkMzVm4NTIyZTkZ...goYzI43Q3ODIyOWM32NjI='
  }
}

// request payment and get a promise that is resolved on payment
var acknowledgement = navigator.payment.request([req]);

// resolve the promise
acknowledgement.then(function(acknowledgement) {
  if(acknowledgement.status === 'processed') {
    // payment was processed by the payment processor
  } else {
    // payment was not processed by the payment processor
  }
});
    A more involved example can be found here:

  }
  paymentCompleteService: 'https://merchant.example.com/services/paymentComplete?transaction=923847298'

}

// resolve the promise (performed in the next section)

  } else {

  }
});

Processing a Payment Acknowledgement

  1. The payment service provider receives a payment request, authenticates the payer, and gets approval to proceed with the payment from the payer.
  2. Once approval has been provided by the payer, the payment is processed
  3. The payment service processor generates a signed payment acknowledgement message, which is a [[!JSON-LD]] object that contains enough information to verify that the transaction has completed in success or failure by the merchant application: 
    var acknowledgement = {
  '@context': 'https://w3id.org/web-payments/v1',
  type: 'PaymentAcknowledgement'
  description: 'Payment to ExampleMerch for widgets"
  status: 'processed',
  scheme: {
    scheme: 'https://w3id.org/payment-schemes#Visa'
    monetaryRequest: {
      amount: '4.35',
      currency: 'USD'
    },
    destination: '20389472398'
  },
  signature: {
    type: 'GraphSignature2015',
    creator: 'https://payment-service-provider.example.com/keys/12',
    created: '2015-09-23T20:23:15Z',
    signatureValue: 'm4NTIyZTOGQzNGVkMzVkZ...OWM32NjIgoYzI43Q3ODIy='
 }
}

// acknowledge that the payment was processed
navigator.payment.acknowledge(acknowledgement);

The Application Programming Interface

This browser-based API is provided for convenience purposes only. The existence of the API is absolutely not a requirement for the basic operation of the Web Payments protocol. No browser API is necessary in order for a Web application to initiate payment and receive an acknowledgement of the result of the payment.

This API provides a clean mechanism that enables developers to initiate payments in a User Agent. A conformant payment agent MUST implement the entirety of the following API.

NavigatorPayment

Navigator Payment is the name of the high-level programming interface that Web developers use to initiate payments. If MUST be made available via the navigator.payment object.

DOMRequest request()

Initiates a payment given an array of [[!JSON-LD]] encoded payment requests describing the type of financial operations to perform.

Should we break refunds out into their own method call? The downside of doing that is that the API is no longer fairly generic. However, .request() doesn't really imply "refund", unless you think of a refund like a reverse payment? The other thing we could do is use .process() or .transact() as the method call and change the interface name to something like navigator.funds.transact() or navigator.wallet.transact().

Each request object will contain a typ key. The value associated with this key will be used to perform matchmaking between the customer's list of registered payment providers and the list of preferred payment providers for the Merchant. Typically, a User Agent will request that the customer rank their registered payment providers in most preferable to least preferable order. If this ranking is performed, a payment provider can be selected automatically by picking highest ranked payment provider in the customer's payment provider list with a payment provider that also exists in the Merchant's payment provider list.

object[] requests
An array of [[!JSON-LD]] encoded payment requests describing the type of financial operations to perform.
